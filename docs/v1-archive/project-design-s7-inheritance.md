# 新项目设计文档 · §7 从 Dickens 继承与革新

> 本节系统性地盘点 Dickens 的所有能力、工具和设计，明确哪些保留（及如何改进）、哪些抛弃（及替代方案）。
> 基于 Dickens 源码（21 个工具、11 个模型、13 个 Agent prompt、12 个知识库）和实际项目（《穿越后宫奇遇记》42 版本迭代经验）的深度分析。

---

## 7.1 继承清单

### ✅ 完整继承（经实践验证有效）

| Dickens 能力 | 墨染对应 | 保留理由 |
|-------------|----------|----------|
| 六阶段工作流 | 墨衡编排的六阶段 | 流程清晰，每阶段职责明确，实践证明有效 |
| 跨模型族审校 | 明镜跨族配置 | 避免同源偏见，是检测 AI 味的基础保障 |
| 分层摘要系统 | 载史三级摘要 | 突破上下文窗口限制的核心机制 |
| 版本化存档 | `chapter_versions` 表 + Git 里程碑 | 42 版本的回溯能力证明了其价值 |
| 持久化通信 | Agent 间通过数据库通信 | Agent 崩溃不丢失数据 |
| 创意简报流程 | 灵犀三阶段 | 发散→聚焦→结晶的方法论经过实践验证 |
| 人物先行方法论 | 匠心角色设计 | 传记法 + 心理画像产出了 23 个有深度的角色 |
| Anti-AI 自检 | 执笔写作后自检 | 第一道防线，虽不完美但必要 |

### ✅ 继承但重大升级

| Dickens 能力 | 问题 | 墨染改进 |
|-------------|------|----------|
| Jaggers 三轮审校 | 漏检"呆板度"，通过标准偏低 | 升级为四轮审校 + Burstiness 硬门槛 + 呆板度检测 + 四层反馈格式 |
| context-builder.ts (1251行) | 硬编码上下文组装逻辑，不够灵活 | 重构为 ContextAssembler 引擎，支持权重配置和按需加载 |
| 知识库 (12个MD文件) | 只读，不支持运行时更新 | CRUD + 版本管理 + 分类体系 + 按需加载策略 |
| 文风指南 (style-guide) | 膨胀到 680 行，打补丁式积累 | 拆分为核心原则(≤500字) + 可插拔专项模块 |
| 章节 Brief | 过于简略或过于详细（两极分化） | 三层 Brief 设计（硬约束/软引导/自由区） |
| Lessons 系统 | 只有事后积累，无预防机制 | 审校驱动自动候选 + 用户确认 + 相关性匹配 + 过期淘汰 |
| 大纲系统 | 弧段级规划不够灵活 | Plantser Pipeline（弧段级规划 + 章节级弹性） |
| 一致性追踪 | 文件系统存储，查询效率低 | PostgreSQL 结构化存储，支持高效查询和统计。伏笔状态机：`planted → developing → resolved \| stale`（见 §3.5 / §6.6） |
| Agent 命名 | 英文名（Micawber/Wemmick/...） | 全部中文命名（灵犀/匠心/...），更直观 |

### ✅ 全新能力（Dickens 完全没有的）

| 墨染能力 | 说明 | 设计位置 |
|----------|------|----------|
| **析典（参考作品分析）** | 基于文学分析理论（叙事学/Enneagram/Sanderson 定律/计算文体学等）对参考作品进行九维深度分析，提炼写作技法沉淀到知识库 | §4.8 |
| **风格子名机制** | 每个风格有诗意子名（执笔·剑心/执笔·云墨等），混合格式配置（YAML + 散文 + 示例段落） | §4.4 |
| **Hybrid 项目管理** | 利用 OpenCode session 管 AI 交互，自建轻量层管小说元数据，消除冗余 | §2.4 |
| **WebUI 七面板** | 新增分析面板（析典报告展示/知识沉淀）和可视化面板（人物关系图/地点树/事件时间线），从五面板扩展为七面板 | §5.3.5/§5.3.7 |

## 7.2 抛弃清单

### ❌ 不继承（经实践证明无效或不适用）

| Dickens 能力 | 抛弃理由 | 墨染替代方案 |
|-------------|----------|------------|
| **多写手独立 Agent** (Weller/Baoyu/Daiyu/Xifeng) | 42 版本迭代证明：写手切换导致上下文断裂、风格跳跃、调试困难 | 单写手（执笔）+ 混合格式风格配置（YAML + 散文 + 示例段落） |
| **固定温度 0.8** | 所有章节用同一温度是质量不稳定的重要原因 | 温度场景化（日常 0.70 ↔ 爆点 0.95） |
| **自由格式审校反馈** | 执笔无法精确定位问题位置和修改方向 | 四层结构化反馈（issue/evidence/suggestion/effect） |
| **Dickens 总指挥直接作为 CLI 入口** | 不支持 WebUI 和 SSE 实时推送 | 墨衡通过 OpenCode SDK 与 WebUI 双向通信 |
| **consistency/ 目录（文件系统）** | 300 万字规模下文件系统无法高效查询伏笔、时间线、角色状态 | PostgreSQL 结构化存储 |
| **两套写作策略并存** (策略A大纲驱动 / 策略B角色驱动) | 实际验证效果都不太好。大纲驱动→呆板，角色驱动→发散 | Plantser Pipeline（融合两者优点） |
| **硬编码的 worldbuilding 子系统** | 实际项目涌现了 11 个子系统，远超固定的 settings/locations | 开放式子系统注册表（用户可自由添加） |
