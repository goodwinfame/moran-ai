# S1 — 概述与定位变更

> 本章说明墨染 V2 的核心定位、与 V1 的关键差异，以及贯穿全系统的设计理念。

---

## 1. 产品定位（不变）

**墨染（MoRan）** 是一款 AI 辅助网文创作工具，核心目标：

- **目标体量**：单部作品 300 万字以上
- **创作模式**：AI 是主要创作者，用户输入想法、做决策、把关质量
- **质量标准**：输出不可识别为 AI 生成，具备商业发表水准
- **覆盖范围**：从灵感到完稿的全流程——脑暴、设定、角色、大纲、写作、审校、归档

---

## 2. V2 核心变化

### 2.1 交互模型重塑

| | V1 | V2 |
|---|---|---|
| **用户面对** | 10 个 Agent 各自的交互界面 | 唯一入口：墨衡对话窗口 |
| **页面结构** | 多页面筹备向导 → 7 面板写作台 | 单页面：左侧聊天 + 右侧信息面板 |
| **操作方式** | 在不同页面间跳转，每个阶段有专属 UI | 全程在同一对话中推进，面板自动切换上下文 |

**为什么改**：

1. **降低认知负担**——用户无需理解 10 个 Agent 的分工，只需与墨衡自然对话
2. **连贯创作体验**——不再因页面跳转打断思路，墨衡记住完整对话历史
3. **灵活流程控制**——用户可以随时回溯、跳转阶段，不受固定向导步骤限制
4. **简化前端复杂度**——从 15+ 页面缩减为 1 个主页面 + 项目列表页

### 2.2 编排层迁移

| | V1 | V2 |
|---|---|---|
| **编排位置** | Hono 后端（Orchestrator 状态机） | OpenCode 内部（墨衡 Agent） |
| **调度方式** | 代码硬编排（route → engine → bridge → OpenCode） | 墨衡通过 SubtaskPart 调度子 Agent |
| **流程保障** | Orchestrator 状态检查 | MCP 工具内置门禁 + 墨衡对话引导 |
| **上下文组装** | Engine 拼接 prompt | 墨衡根据 MCP 查询结果组装上下文 |

**为什么改**：

1. **消除中间层**——V1 的 Hono Engine → Bridge → OpenCode 链路过长，每层都需要维护状态
2. **利用 LLM 判断力**——流程分支决策（如审校发现设定问题，该修设定还是修文本）交给墨衡判断，比硬编码状态机更灵活
3. **工具直连 DB**——子 Agent 通过 MCP 工具直接读写数据库，不再依赖 Engine 解析 LLM 文本输出

### 2.3 Session 架构简化

| | V1 | V2 |
|---|---|---|
| **Session 数量** | 每个 Agent 独立 Session | 每个项目一个墨衡 Session |
| **消息持久化** | Zustand 内存（刷新丢失） | OpenCode 自动持久化（`session.messages()`） |
| **会话分叉** | 不支持 | 支持（`session.fork()` 用于子 Agent 工作） |

## 3. 设计理念

### 3.1 从 V1 继承的 10 条核心理念

来自 Dickens 经验（5 条）：

1. **Memory-Slice 分层存储**——六大类别（guidance / world / characters / consistency / summaries / outline）三层温度管理
2. **ManagedWrite 管线**——所有写入经过验证 → 分类 → 增量化 → 路由 → 背压，防止记忆膨胀
3. **ContextAssembler 按需装配**——根据场景分析结果动态分配 token 预算
4. **四轮审校不妥协**——AI 味检测（Burstiness ≥ 0.3）→ 逻辑一致性 → 文学质量（RUBRIC ≥ 7.5）→ 读者体验
5. **Lessons 自学习**——从用户修正中提取经验，未来章节自动规避同类问题

墨染新增（5 条）：

6. **Plantser Pipeline**——硬约束 / 软引导 / 自由区三层 Brief，平衡规划与即兴
7. **角色五维心理模型**——GHOST（创伤根源）→ WOUND（心理伤痕）→ LIE（核心谎言）→ WANT（表层欲望）↔ NEED（真实需求）
8. **风格引擎混合格式**——YAML 约束 + 散文描述 + 示例段落，确保风格既可量化又可感知
9. **析典九维分析**——叙事结构 / 角色设计 / 世界观 / 伏笔 / 节奏张力 / 爽感机制 / 文风指纹 / 对话声音 / 章末钩子
10. **螺旋检测**——审校螺旋（反复不过）、膨胀螺旋（记忆无限增长）、矛盾螺旋（设定冲突累积）

### 3.2 V2 新增理念

11. **单入口对话式**——用户只与墨衡交互，其余 Agent 是墨衡的"内部团队"，用户无感知
12. **MCP 工具门禁**——每个工具调用时检查前置条件（如写章节前必须有大纲），门禁逻辑在工具内部实现，不依赖外部状态机
13. **Agent 通过 DB 通信**——子 Agent 之间不直接传递消息，所有产出写入 DB，其他 Agent 通过 MCP 工具查询获取
14. **面板即视图**——信息面板是 DB 数据的实时只读视图，不持有业务状态，数据变更由 Agent 通过 MCP 驱动

## 4. 设计目标（更新）

继承 V1 的 8 项设计目标，标注 V2 变化：

| 编号 | 目标 | V2 变化 |
|------|------|---------|
| G1 | 300 万字不退化 | 不变——UNM 记忆引擎完整保留 |
| G2 | AI 文本不可识别 | 不变——四轮审校 + Burstiness 检测保留 |
| G3 | 角色一致性 | 不变——五维心理模型 + 关系网络保留 |
| G4 | 世界观自洽 | 不变——世界设定子系统 + 自洽检查保留 |
| G5 | 风格可控 | 不变——风格引擎 + 9 种预设保留 |
| G6 | 流程可回溯 | **增强**——对话历史天然可回溯，版本管理保留 |
| G7 | 成本可控 | 不变——token 预算 + 成本追踪保留 |
| G8 | 开发者友好 | **增强**——MCP 工具标准化，新增工具只需实现 MCP 接口 |

## 5. 目标用户

**不变**：有想法但缺乏时间/精力将想法完整落地的网文创作者。

用户画像：
- 有故事灵感，但难以坚持日更
- 理解网文套路，但需要 AI 承担大量执行工作
- 愿意做决策者和质量把关者，而非逐字逐句写作者
- 技术素养：会使用对话式 AI 工具（类 ChatGPT 交互经验）
